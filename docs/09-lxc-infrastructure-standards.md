# LXC 基础设施规范

本文档定义在 Proxmox VE (PVE) 上使用无特权 LXC 容器部署基础服务的标准规范，适用于家庭实验室（homelab）及中小型自托管场景。

---

## 一、LXC 创建通用规范

| 参数 | 推荐值 | 说明 |
|------|--------|------|
| 类型 | LXC | 非 KVM 虚拟机 |
| 特权模式 | **无特权（unprivileged）** | 默认选择，安全性更好 |
| 嵌套虚拟化 | **不开启** | 基础服务不需要 |
| 模板 | **Debian 12** | 稳定、软件包丰富 |
| 开机自启 | **开启** | 宿主重启后服务自动恢复 |
| 备份 | **开启** | 纳入 PVE 备份任务 |

---

## 二、数据目录规范

所有服务的持久化数据目录统一使用 `/data/<service>` 格式，通过 PVE 挂载点 `mp0` 挂载专用数据卷。

| 服务 | 容器内数据目录 |
|------|--------------|
| MySQL | `/data/mysql` |
| Redis | `/data/redis` |
| MinIO | `/data/minio` |
| Gitea | `/data/gitea` |
| Prometheus | `/data/prometheus` |
| Grafana | `/data/grafana` |
| Loki | `/data/loki` |

> **重要：** 不要将 `mp0` 直接挂载到 `/var/lib/<service>`，统一使用 `/data/<service>` 并在服务配置中显式指定数据目录。这样结构更清晰，便于备份、迁移和运维。

---

## 三、存储挂载规范

每个服务容器使用两块存储：

- **rootfs**：仅存放操作系统和软件包，不存放业务数据
- **mp0**：专用数据卷，挂载到容器内 `/data/<service>`

### 挂载点配置建议

| 参数 | 推荐值 |
|------|--------|
| 挂载点 ID | `0`（即 mp0） |
| 存储池 | 选择你的业务数据存储池 |
| 路径（容器内） | `/data/<service>` |
| 备份 | **勾选** |
| 只读 | **不勾选** |
| 跳过复制 | 一般不勾选 |
| ACLs | 默认 |

## 五、基础服务资源规格建议

| 服务 | CPU (vCPU) | 内存 | rootfs | 数据卷 (mp0) |

## 六、网络规范

- 所有基础服务容器分配**固定内网 IP**
- DNS 指向局域网 DNS / 路由器 / 公共 DNS
- 基础服务不直接暴露公网端口，通过反向代理统一对外

---

## 七、权限注意事项（无特权 LXC）

无特权 LXC 使用 UID/GID 映射，容器内 `root` 并非宿主机 `root`。挂载 `mp0` 后，服务进程可能因目录权限不对无法启动。

### 标准处理步骤

1. 创建容器并挂载 `mp0` 到 `/data/<service>`
2. 进入容器，安装服务
3. 停止服务
4. 检查数据目录属主：`ls -ld /data/<service>`
5. 修正属主：`chown -R <service-user>:<service-group> /data/<service>`
6. 修改服务配置，将数据目录指向 `/data/<service>`
7. 启动服务并验证

---

## 八、部署顺序建议

建议按以下顺序部署，从轻到重逐步验证：

1. `redis-01`（轻量，验证权限模型）
2. `mysql-01`（数据库，验证 datadir 迁移）
3. `minio-01`（对象存储）
4. `gitea-01`（代码管理）
5. `monitor-01`（监控告警）
